# Technical Product Requirements Document (PRD)

## 1. Product Overview
**Project Name:** HealthAI (Production Code Name)
**Platform:** Cross-platform Mobile (iOS & Android)
**Framework:** Flutter (Dart)
**Core AI Engine:** Google Gemini API (Pro & Flash models)
**Status:** Planning Phase / Pre-Production

**Purpose:** 
HealthAI is a comprehensive, AI-native health, fitness, diet, and habit-tracking application. Unlike traditional trackers that rely heavily on manual data entry and static logic, HealthAI leverages the **Google Gemini API** to provide dynamic, conversational, and context-aware health coaching, multi-modal food logging, and personalized habit formulation.

---

## 2. Technical Architecture & Stack

### 2.1 Frontend
- **Framework:** Flutter (latest stable version)
- **Language:** Dart
- **State Management:** Riverpod (for robust, scalable, and compile-safe dependency injection and state management).
- **Local Database:** Isar (NoSQL) or SQLite (sqflite) for fast, offline-first data caching (workouts, past meals, habit streaks).
- **Navigation:** GoRouter (for declarative routing and deep linking support).
- **UI/UX:** Material 3 & Cupertino widgets, customized with a premium dual-theme design system. Heavy use of `flutter_animate` for fluid micro-interactions. The app features a **Native-Grade Light and Dark Mode Engine**, dynamically switching based on OS preferences, with deeply customized, hand-picked color tokens for both environments rather than relying on automatic color inversion.

### 2.2 Backend Architecture
We utilize a **Serverless & Managed Backend** approach for optimal scaling, security, and low maintenance:
- **Backend-as-a-Service (BaaS):** Firebase (Google Cloud).
- **Authentication:** Firebase Auth (Email/Password, Google, Sign in with Apple).
- **API Engine (Middle-Tier):** Firebase Cloud Functions (Node.js/TypeScript). 
  - **Purpose:** Acts as a secure proxy between the Flutter client and the Gemini API. This ensures API keys are strictly hidden in backend environment variables and never exposed to the client app.
  - **Logic:** Performs prompt orchestration, sanitizes user inputs, and enforces rate limits before hitting Gemini.
- **Media Storage:** Firebase Cloud Storage (Used for temporarily uploading food images to be fed into Gemini Vision, and for persistent profile/progress photos).
- **Notifications:** Firebase Cloud Messaging (FCM) + `flutter_local_notifications` for on-device reminders (water, habits).

## 3. Database Structure & Schema (Firestore / NoSQL)

We will use **Cloud Firestore**, a scalable NoSQL document database, suited for real-time syncing and offline-first mobile architectures.

### 3.1 `Users` Collection (`users/{userId}`)
Stores core profile data, preferences, and the summarized context for Gemini.
```json
{
  "uid": "abc123xyz",
  "email": "user@example.com",
  "display_name": "Jane Doe",
  "dob": "1990-05-15",
  "gender": "female",
  "height_cm": 165,
  "weight_kg": 68,
  "created_at": "Timestamp",
  "last_active": "Timestamp",
  "preferences": {
    "dietary": ["Vegan", "Gluten-Free"],
    "unit_system": "metric",
    "theme": "system"
  },
  "gemini_context_summary": "User is a 34yo vegan aiming for 5kg weight loss. Prefers home workouts." 
}
```

### 3.2 `DailyLogs` Subcollection (`users/{userId}/daily_logs/{date_YYMMDD}`)
Aggregates all daily activity into a single document for fast read/writes and simplified queries for the dashboard.
```json
{
  "date": "2026-02-24",
  "water_ml": 1500,
  "calories_consumed": 1850,
  "calories_burned": 400,
  "macros": { 
    "protein": 80, 
    "carbs": 200, 
    "fat": 50 
  },
  "sleep_minutes": 420,  // Synced via HealthKit/Health Connect
  "step_count": 8500
}
```

### 3.3 `Meals` Subcollection (`users/{userId}/meals/{mealId}`)
Individual meal entries, often generated by Gemini analyzing an image.
```json
{
  "meal_id": "meal_890",
  "date_logged": "Timestamp",
  "meal_type": "Lunch",
  "name": "Avocado Toast with Poached Egg",
  "image_url": "gs://bucket/path/to/image.jpg", // Optional
  "calories": 450,
  "macros": { "protein": 15, "carbs": 35, "fat": 25 },
  "ingredients_detected": ["Avocado", "Egg", "Sourdough Bread", "Chili Flakes"],
  "ai_generated": true
}
```

### 3.4 `Workouts` Subcollection (`users/{userId}/workouts/{workoutId}`)
Detailed logs of completed workouts or structured workout plans generated by Gemini.
```json
{
  "workout_id": "wk_001",
  "date": "Timestamp",
  "title": "Upper Body Hypertrophy",
  "duration_seconds": 2700,
  "source": "ai_generated", // or "manual", "pre_built"
  "exercises": [
    {
      "name": "Dumbbell Bench Press",
      "sets": [
        {"reps": 10, "weight_kg": 20, "completed": true},
        {"reps": 8, "weight_kg": 22.5, "completed": true}
      ]
    },
    // ... more exercises
  ]
}
```

### 3.5 `Habits` Subcollection (`users/{userId}/habits/{habitId}`)
Tracks the custom habits and their current streaks.
```json
{
  "habit_id": "hab_12",
  "title": "Read 10 Pages",
  "frequency": "daily",
  "time_of_day": "evening",
  "current_streak": 5,
  "longest_streak": 14,
  "created_at": "Timestamp",
  "active": true
}
```

## 4. Core Features & Technical Implementation

### 4.1 AI-Powered Diet & Nutrition (Multimodal Gemini)
**Feature:** Users log food by taking a picture, describing a meal via voice, or typing. 
**Technical Flow:**
1. Image captured via Flutter `camera` plugin.
2. Image uploaded to Firebase Cloud Storage.
3. Payload (Storage URI or Base64) sent to secure Cloud Function.
4. Cloud Function calls **Gemini API (Multimodal - 1.5 Flash)** with a highly optimized prompt: *"Analyze this food image. Return a strict JSON object containing: meal_name, estimated_calories, protein_g, carbs_g, fat_g, and ingredients_list."*
5. App parses the JSON response, presents it to the user for confirmation, and saves the new document to the `Meals` collection and updates the `DailyLogs` totals.

### 4.2 Dynamic AI Health Coach (Conversational Gemini)
**Feature:** A 24/7 intelligent chat interface for health advice, workout adjustments, and motivation.
**Technical Flow:**
1. **Context Aggregation:** Upon opening chat, the app pulls the user's `preferences`, `gemini_context_summary`, and the last 7 documents from `DailyLogs`.
2. **System Prompting:** The Cloud Function injects this data into the Gemini prompt: *"You are HealthAI. The user consumed 1800 cals today. They ask: 'Should I work out?'..."*
3. **Streaming Responses:** Uses Gemini's streaming API via the Cloud Function passing chunks back to Flutter so the UI updates in real-time.

### 4.3 Workout & Fitness Engine
**Feature:** Customized workout generation, real-time logging, and rest timers.
**Technical Flow:**
1. User inputs goals.
2. Cloud Function calls Gemini 1.5 Pro to generate a structured JSON workout plan.
3. App natively renders this JSON from Firestore into interactive workout screens.

### 4.4 Habit & Hydration Tracking
**Technical Flow:**
1. **Hydration Logic:** Base water requirement calculated on-device.
2. **Cloud Adjustment:** Scheduled Cloud Function uses weather APIs + Gemini to optionally suggest adjusting the daily target.
3. **Local Notifications:** App schedules native OS local notifications so reminders work offline.

## 5. Security & Privacy
- **API Key Protection:** Google Gemini API keys are *never* shipped in the Flutter client code. All AI calls route strictly through Firebase Cloud Functions.
- **Data Privacy:** Health data retrieved from HealthKit/Health Connect is kept entirely on-device unless explicitly authorized by the user to be sent to the Gemini AI for analysis. Data is anonymized before LLM processing.

## 6. QA & Testing Strategy
- **Unit Testing:** Focus on business logic handlers, state notifiers (Riverpod), and date/time manipulation logic.
- **Widget Testing:** Verify UI states (loading, error, success) for custom components (charts, habit trackers).
- **Integration Testing:** `integration_test` package for testing core user flows (Onboarding -> Take photo of food -> Save log).
